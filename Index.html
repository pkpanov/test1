SECTION 00

The mental model first

Before any code, internalize the distinction that governs everything else.

  • Immediate-mode rendering — what raw canvas gives you. You call ctx.fillRect(...) and pixels appear. The canvas remembers nothing; there are no "objects" on it.
  • Retained-mode rendering — what the DOM and SVG give you. You describe objects; the system keeps them and redraws for you.

A board with custom shapes, animation and interactivity needs the benefits of retained mode sitting on top of an immediate-mode surface. So the core job of your library is simple to state:

The whole idea

Keep a model of the scene in JS, and re-render that model to the canvas every frame. State → render loop → pixels → input → back to state. Get this loop right and everything else is detail.

SECTION 01

What makes a JavaScript component reusable

"Reusable" has a concrete test: a stranger can drop your component into a project you've never seen, configure it for their case, run several copies on one page, and remove it cleanly — all without reading your source. Five properties get you there.

  1. Encapsulation. Internal state is private so consumers can't reach in and break your invariants. JavaScript gives you several ways to hide state — closures, #private fields, WeakMap, or a leading-underscore convention (see §02).
  2. A small, explicit contract. A handful of methods, events and options. Everything else is implementation detail you stay free to change without breaking anyone.
  3. No hidden dependencies. It must not assume a global, a specific element id, a framework, or that it's the only instance on the page. Pass in what it needs — this is dependency injection.
  4. Configurable, with sane defaults. Useful at zero config; bends to many cases through options and extension points (see §13).
  5. An explicit lifecycle. create and destroy so it can be spun up and torn down repeatedly with no leaks (see §21).

The one mistake that quietly kills reuse: shared state

Module-level mutable state is shared by every "instance," so two boards on one page corrupt each other. State must live per instance.

// ❌ module-level state — every "board" shares one array
let shapes = [];
export function addShape(s) { shapes.push(s); }

// ✅ each board owns its own state
export function createBoard(canvas) {
  const shapes = [];                 // private & per-instance
  return { add: (s) => shapes.push(s) };
}

Inject dependencies; don't reach for globals

// ❌ hardcoded global lookup — one instance only, untestable, brittle
const canvas = document.getElementById('board');

// ✅ the caller supplies the surface (and, for tests, even the clock)
class DrawBoard {
  constructor(target, { now = () => performance.now() } = {}) {
    this.canvas = typeof target === 'string'
      ? document.querySelector(target)   // accept a selector…
      : target;                          // …or a node the caller already has
    this.now = now;                      // injected clock = deterministic tests
  }
}
Reuse anti-patterns

Side effects at import time, hardcoded selectors, reading or writing window.*, throwing on missing optional config, and assuming a single instance. Each one is a tripwire for the next person who tries to reuse your code.

SECTION 02

Alternative approaches in JavaScript

There's no single "right" way to author a component in JS. The language offers several, each with different ergonomics around privacy, this, inheritance and how consumers use it. Here are the eight worth knowing, with when to pick each.

1 · ES Class

export class DrawBoard {
  #shapes = [];                       // hard-private field
  static from(json) { /* named constructor */ }
  get count() { return this.#shapes.length; }
  add = (s) => { this.#shapes.push(s); return this; };  // arrow field → `this` is bound
}

Discoverable, great tooling, real inheritance and instanceof. The arrow-field trick sidesteps the classic this-binding bug when a method is passed as a callback. The default recommendation.

2 · Factory function + closures

export function createDrawBoard(canvas, opts = {}) {
  const shapes = [];                  // private by closure — no #, no this
  const api = {
    add(s) { shapes.push(s); return api; },
    get count() { return shapes.length; },
  };
  return Object.freeze(api);          // optionally lock the public surface
}

Airtight privacy, no this, and you return only the public API. Methods are re-created per instance (negligible cost); no instanceof. Ideal for a functional style.

3 · Module / revealing-module pattern (IIFE)

const BoardKit = (function() {
  let registered = {};                // private
  function register(name, fn) { registered[name] = fn; }
  return { register };               // reveal a subset
})();

The pre-ESM classic. It naturally produces a singleton, which is great for a shared registry or a global <script> build, but wrong when you need many instances — use a factory or class for those. UMD bundles still use this shape to expose a global.

4 · Web Component (Custom Element + Shadow DOM)

class DrawBoardEl extends HTMLElement {
  static observedAttributes = ['theme'];
  #board;
  connectedCallback() {
    const root = this.attachShadow({ mode: 'open' });   // style isolation
    const canvas = document.createElement('canvas');
    root.append(canvas);
    this.#board = new DrawBoard(canvas, { theme: this.getAttribute('theme') });
  }
  attributeChangedCallback(name, _old, val) { this.#board?.set(name, val); }
  disconnectedCallback() { this.#board?.destroy(); }   // automatic cleanup
}
customElements.define('draw-board', DrawBoardEl);

Usable as plain HTML (<draw-board theme="dark"></draw-board>), framework-agnostic, style-encapsulated, with a lifecycle the platform runs for you. Costs: more ceremony, string-only attributes, and SSR/hydration nuances. The strongest "reuse everywhere" option (more in §15).

5 · Prototype + Object.create

const BoardProto = {
  add(s) { this.shapes.push(s); return this; },
};
function makeBoard() {
  const b = Object.create(BoardProto);   // methods shared on the prototype
  b.shapes = []; return b;
}

The mechanism classes are built on. Rarely hand-written today, but worth understanding: methods live once on the prototype (memory-friendly), and it's the basis of composition below.

6 · Composition over inheritance (mixins)

// each mixin adds one capability; compose the ones you want
const withEvents  = (o) => Object.assign(o, { on() {}, off() {}, emit() {} });
const withHistory = (o) => Object.assign(o, { undo() {}, redo() {} });

const board = withHistory(withEvents(createCore(canvas)));

Instead of a deep class tree (Board extends Interactive extends Drawable…), assemble capabilities à la carte. This scales best for a feature-rich component — features become opt-in pieces rather than rigid hierarchy.

7 · Extend EventTarget (platform pub-sub)

class DrawBoard extends EventTarget {
  add(s) {
    /* … */
    this.dispatchEvent(new CustomEvent('change', { detail: { shape: s } }));
  }
}
board.addEventListener('change', (e) => console.log(e.detail.shape));

Inherit the browser's own event machinery — consumers use the same addEventListener they already know. Data rides in event.detail.

8 · Plugin architecture

const board = new DrawBoard(canvas);
board.use(GridPlugin).use(SnapPlugin, { grid: 8 });

// a plugin is just an installer
const GridPlugin = { install(board, opts) { /* extend board here */ } };

Keep the core tiny and ship features as plugins users opt into. Best for bundle size and third-party extensibility.

SECTION 03

Module systems & build outputs

How you author and publish decides who can consume you. Author in ESM, then emit several formats from that one source.

  • ESM — import/export; statically analyzable, so bundlers can tree-shake. The modern default and what you write.
  • CommonJS (CJS) — require/module.exports; older Node and some tooling.
  • UMD — one bundle that works as a global, CJS, or AMD; friendly to CDNs and <script>.
  • IIFE / global — a single script that hangs window.DrawBoard; the simplest drop-in for a plain HTML page.
package.json — point each consumer at the right build
{
  "name": "draw-board",
  "type": "module",
  "exports": {
    ".": {
      "import": "./dist/draw-board.js",     // ESM consumers
      "require": "./dist/draw-board.cjs"     // CJS consumers
    }
  },
  "unpkg": "./dist/draw-board.umd.js",        // CDN <script> tag
  "types": "./dist/index.d.ts",               // editor autocomplete
  "sideEffects": false                        // safe to tree-shake
}
Authoring rules for tree-shaking

Prefer named exports over a single default object, keep the module body free of top-level side effects, and split optional features (extra shapes, tools) into their own modules so unused code can be dropped. Ship source maps.

SECTION 06

Coordinates, transforms & a camera

A real board needs pan and zoom. The clean way is a camera: one transform applied before drawing the scene, plus its inverse to turn pointer coordinates back into world space.

class Camera {
  x = 0; y = 0; zoom = 1;
  apply(ctx) {                       // screen = world * zoom + offset
    ctx.translate(this.x, this.y);
    ctx.scale(this.zoom, this.zoom);
  }
  toWorld(sx, sy) {                  // invert it for hit-testing
    return { x: (sx - this.x) / this.zoom,
             y: (sy - this.y) / this.zoom };
  }
}

Each frame: ctx.save() → camera.apply(ctx) → draw scene → ctx.restore(). To zoom toward the cursor, convert the cursor to world space before changing zoom, then nudge camera.x/y so the same world point lands under the cursor again.

Trick

Wrap every shape's drawing in ctx.save() / ctx.restore() — it isolates style and transform changes so one shape can't leak state into the next.

The #1 mistake

Never set img.src inside the render loop — assigning src triggers an async load and decode, so you'll flash blanks and thrash memory. Load every image once up front; in the loop you only ever drawImage an already-decoded one.

Going faster

For heavy playback use createImageBitmap(blobOrImage) — it returns a pre-decoded, GPU-friendly bitmap that draws faster than an <img> and can be handed to a Web Worker with OffscreenCanvas. Cap how many frames you keep decoded if you have hundreds.

    var pl=document.getElementById('tweenPlay'); if(pl) pl.addEventListener('click',function(){ if(reduce){drawStatic(1);return;} tt=0; playing=true; last=performance.now(); raf=requestAnimationFrame(frame); });
    v.onResize=function(){ drawStatic(playing?tt:1); };
    whenVisible(canvas, function(){ v.refit(); drawStatic(1); }, function(){ playing=false; cancelAnimationFrame(raf); });
  });
export const easings = {
  linear:        t => t,
  easeOutCubic:  t => 1 - Math.pow(1 - t, 3),
  easeInOutQuad: t => t < .5 ? 2*t*t : 1 - Math.pow(-2*t + 2, 2)/2,
};

class Tween {
  constructor(target, to, { duration = .3, easing = easings.easeOutCubic } = {}) {
    this.target = target; this.to = to; this.from = {};
    for (const k in to) this.from[k] = target[k];
    this.duration = duration; this.easing = easing; this.t = 0;
  }
  update(dt) {
    this.t = Math.min(this.t + dt / this.duration, 1);
    const e = this.easing(this.t);
    for (const k in this.to)
      this.target[k] = this.from[k] + (this.to[k] - this.from[k]) * e;
    this.done = this.t >= 1;
  }
}

The whole trick: mutate a shape's properties over time, mark dirty, let the loop draw it. The same loop drives physics — integrate velocity by dt for inertia after a drag.

class DrawBoard extends Emitter {
  add(shape)              { this.scene.add(shape); this._changed('add', shape); return this; }
  remove(shape)           { this.scene.remove(shape); this._changed('remove', shape); return this; }

      SECTION 17
      
Undo & redo (command pattern)

      
Don't snapshot the whole scene per change — model each edit as a command that applies and reverses itself.

      
class History {
  #past = []; #future = [];
  do(cmd) { cmd.apply(); this.#past.push(cmd); this.#future.length = 0; }
  undo()  { const c = this.#past.pop();   if (c) { c.revert(); this.#future.push(c); } }
  redo()  { const c = this.#future.pop(); if (c) { c.apply();  this.#past.push(c); } }
}

const addShape = (scene, shape) => ({
  apply()  { scene.add(shape); },
  revert() { scene.remove(shape); },
});

      
Tiny memory footprint, and complex edits compose — a "move" stores old and new positions; a multi-select drag is one command wrapping many. The console in §14 uses exactly this.

Avoid GC churn
Don't allocate objects or arrays inside tick(). No .map/.filter per frame, no fresh vectors — reuse them. Per-frame garbage is what the collector sweeps, and that's the visible stutter.

Testing the component

      
Because the engine is split from the DOM (the whole point of §01), most of it tests headlessly — no browser, fast and deterministic.

      SECTION 23
      
Gotchas & pro tips

      
    
        
  • Never set img.src in the loop — preload, then only drawImage (see §11).
    • 
        
  • Canvas text is expensive and unselectable. Cache measureText results; overlay real DOM for editable text.
    • 
        
  • Sub-pixel lines blur. A 1px stroke at integer x sits between pixels — offset by 0.5.
    • 
        
  • imageSmoothingEnabled = false for crisp pixel-art upscaling.
    • 
        
  • Throttle pointermove. Collect points, process once per tick.
    • 
        
  • Clear in the right space. clearRect respects the current transform — reset it or clear in world coords if a camera is applied.
    • 
        
  • Cap the pixel ratio (min(dpr, 2)) on big canvases to avoid huge backing stores on 3× phones.
    • 
        
  • Clear the held-key Set on blur so keys don't stick after alt-tab.
    • 
        
  • Don't mutate the options object the caller passed in — clone it (see §13).
